# @eventaat/api

NestJS + Prisma backend for eventaat. PostgreSQL is the database.

**Step 3 deliverable:** first Prisma model (`User` + `Role` enum), `PrismaModule`
re-enabled, first migration ready to run, and `/health` reporting database status.

Still no business endpoints, no auth, no restaurants/reservations.

## Folder layout

```
apps/api/
├── prisma/
│   ├── schema.prisma         # Role enum + User model
│   └── migrations/           # generated by `prisma migrate dev`
├── src/
│   ├── main.ts               # bootstrap — listens on PORT (default 4000)
│   ├── app.module.ts         # ConfigModule + PrismaModule + HealthModule
│   ├── health/
│   │   ├── health.controller.ts   # GET /health — includes db status
│   │   └── health.module.ts
│   └── prisma/
│       ├── prisma.service.ts # extends PrismaClient + isHealthy() probe
│       └── prisma.module.ts  # @Global() — PrismaService available app-wide
├── .env.example
├── nest-cli.json
├── tsconfig.json
└── tsconfig.build.json
```

## Local setup

Run from the **monorepo root** (`/Users/amjadmohammed/Documents/Claude/Projects/eventaat`):

```bash
# 1. Install all workspace dependencies
npm install

# 2. Copy the env template
cp apps/api/.env.example apps/api/.env

# 3. Start a local PostgreSQL (skip if you already have one)
docker run --name eventaat-pg \
  -e POSTGRES_USER=eventaat \
  -e POSTGRES_PASSWORD=eventaat \
  -e POSTGRES_DB=eventaat \
  -p 5432:5432 -d postgres:16

# 4. Create the first migration and apply it
npm run prisma:migrate -w @eventaat/api -- --name init_user
# This runs `prisma migrate dev`, which:
#   - creates apps/api/prisma/migrations/<timestamp>_init_user/migration.sql
#   - applies it to the database in DATABASE_URL
#   - regenerates the Prisma client (so the User model is typed)

# 5. Start the API
npm run dev -w @eventaat/api
```

## Verify the migration worked

Any of these confirms it:

```bash
# A. The migration directory now exists
ls apps/api/prisma/migrations/
# → should contain something like 20260425103000_init_user/migration.sql

# B. Inspect the table directly
docker exec -it eventaat-pg psql -U eventaat -d eventaat -c "\d users"
# → should print the columns: id (uuid), email, password_hash, full_name,
#                             phone, role (Role enum), is_active, created_at, updated_at

# C. Open Prisma Studio to browse the (empty) users table
npm run prisma:studio -w @eventaat/api
```

## Test the health endpoint

```bash
curl http://localhost:4000/health
```

When the database is up:

```json
{
  "status": "ok",
  "service": "eventaat-api",
  "version": "0.0.1",
  "database": "ok",
  "timestamp": "2026-04-25T12:34:56.789Z",
  "uptimeSeconds": 5
}
```

To prove the graceful failure path, stop the database and curl again:

```bash
docker stop eventaat-pg
curl http://localhost:4000/health
```

```json
{
  "status": "degraded",
  "service": "eventaat-api",
  "version": "0.0.1",
  "database": "unavailable",
  "timestamp": "2026-04-25T12:35:21.123Z",
  "uptimeSeconds": 30
}
```

The API does **not** crash — `PrismaService.isHealthy()` swallows the error and
the controller reports `database: "unavailable"`. Restart Postgres
(`docker start eventaat-pg`) and the next `/health` call returns to `"ok"`.

## Scripts

| Script              | What it does                                                    |
|---------------------|-----------------------------------------------------------------|
| `dev`               | Build `@eventaat/shared`, then run NestJS in watch mode         |
| `build`             | Build `@eventaat/shared`, then `nest build` to `dist/`          |
| `start`             | Run the compiled server (`node dist/main.js`)                   |
| `typecheck`         | `tsc --noEmit`                                                  |
| `prisma:generate`   | Regenerate the Prisma client from `schema.prisma`               |
| `prisma:migrate`    | Run `prisma migrate dev` (creates + applies migrations)         |
| `prisma:studio`     | Open Prisma Studio (DB GUI)                                     |
